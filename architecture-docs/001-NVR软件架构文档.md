# 001《NVR 软件架构文档》

## 1. 文档目标

本文档用于收敛当前 NVR 项目的正式软件架构方向，作为后续详细设计、代码重构、HAL 接口抽取、功能扩展与 SoC 验证的统一上层依据。

本文档不以当前 UI demo 的代码结构为准，而以当前项目的真实目标、约束和后续演进需要为准。

它的直接用途包括：

- 指导当前 demo 向正式产品架构迁移
- 指导多 SoC 兼容的软件边界设计
- 指导私有多媒体 SDK 的上层抽象方式
- 指导页面、服务、状态、HAL 的职责划分
- 指导后续 HAL 层接口文档设计
- 指导 SoC 验证点和诊断能力设计
- 为没有独立开发能力的下游厂商提供可承接的应用平台基础

---

## 2. 项目背景与定位

### 2.1 项目背景

当前 NVR 项目运行环境主要基于：

- Linux 4.9 或 Linux 5.10
- 私有多媒体 SDK
- 多媒体链路整体参考海思平台思路
- 不同 SoC 在能力、命名和实现细节上存在差异

你们不是单纯做一个固定机型的 NVR，而是在 SoC 设计公司主导下做一套：

- 可跨多颗不同性能 SoC 运行的 NVR 平台
- 可为 SoC 能力开发与测试服务的应用系统
- 可为下游缺乏独立开发能力的厂商提供成套应用支持的产品基础

并且，当前项目的运行结果还可能反向影响下一代 SoC 的设计和 SDK 规划。

### 2.2 项目定位

因此，这个 NVR 项目本质上不是普通的“终端 UI 工程”，而是一个兼具以下角色的平台：

1. **产品交付平台**
   - 对外提供完整可运行的 NVR 能力
2. **SoC 验证平台**
   - 暴露编解码、显示、存储、网络、AI 等能力边界
3. **平台演进平台**
   - 为后续新 SoC、新 SDK、新功能提供稳定承载骨架
4. **参考实现平台**
   - 为下游客户提供可移植、可裁剪、可扩展的应用支持

这决定了架构必须同时满足：

- 可维护
- 可扩展
- 可裁剪
- 可诊断
- 可适配
- 可验证

---

## 3. 架构设计目标

结合项目背景，正式 NVR 软件架构应满足以下目标。

### 3.1 支持多 SoC 兼容

系统应能够：

- 在不同性能、不同能力边界的 SoC 上运行
- 支持不同 SDK 版本和不同命名习惯
- 将 SoC 差异收口到平台适配层，而不是扩散到 UI 和业务层

### 3.2 对私有 SDK 做稳定抽象

系统不能让业务层和 UI 层直接依赖私有 SDK 细节。

必须形成：

- 统一能力查询
- 统一媒体控制语义
- 统一错误码/状态语义
- 统一资源生命周期管理

### 3.3 能支撑正式产品，而不是只支撑 demo

架构应支持：

- 预览
- 回放
- 检索
- 录像
- 事件/告警
- 存储管理
- 配置管理
- 用户与权限
- 诊断与运维
- 后续 AI/联动/扩展能力

### 3.4 能服务 SoC 验证与问题定位

架构必须能明确回答：

- 某功能失败是 UI 问题、服务问题还是 SDK/HAL 问题
- 某 SoC 上功能退化发生在哪里
- 某能力缺失是设计限制、SDK 限制还是资源限制

### 3.5 能支持后续重构与下游交付

正式架构不应依赖当前 demo 的页面拼装方式，而应具备：

- 模块化结构
- 明确服务边界
- 可替换 UI 皮肤/页面实现
- 可裁剪能力组合
- 可沉淀为下游可承接的平台骨架

---

## 4. 总体架构结论

当前项目**不适合继续以 UI demo 的页面驱动方式长大**。

正式 NVR 架构应采用：

**UI 展示层 + 页面控制/应用编排层 + 核心服务层 + 平台/HAL 适配层**

并以以下三个中心作为系统骨架：

1. **统一资源模型**
2. **统一状态模型**
3. **统一事件机制**

同时必须显式引入：

- 能力探测机制
- 功能裁剪机制
- 降级执行机制
- 通道级隔离机制
- 诊断与日志统一机制

这比直接按页面、按钮、菜单堆代码更适合当前项目的真实目标。

---

## 5. 系统边界

建议将正式 NVR 系统边界定义为以下九个能力域：

1. **实时预览域**
2. **录像与回放域**
3. **检索与索引域**
4. **事件与告警域**
5. **设备与通道管理域**
6. **存储管理域**
7. **系统配置域**
8. **用户与安全域**
9. **诊断与运维域**

这九个域不是页面划分，而是系统能力划分。

UI 页面可以映射到这些域，但系统实现不能被页面层次反向绑死。

---

## 6. 分层架构设计

## 6.1 UI 展示层

职责：

- 页面显示
- 控件交互
- 菜单/弹窗/对话框
- 焦点、布局、主题、视觉反馈

原则：

- 不直接访问 SDK/HAL
- 不直接管理复杂业务状态
- 不直接持有长期媒体资源生命周期

典型元素：

- Screen / View
- Widget / Component
- Dialog / Menu
- StatusBar
- VideoViewport

## 6.2 页面控制与应用编排层

职责：

- 将 UI 动作翻译为系统用例
- 组织跨服务流程
- 管理页面状态与异步状态切换
- 将服务层输出转换为 UI 可消费的状态

典型动作：

- OpenPreview(channelId)
- SwitchLayout(layoutMode)
- SearchRecord(query)
- StartPlayback(sessionArgs)
- SaveConfig(section, payload)
- AcknowledgeAlarm(alarmId)
- ExportRecord(exportArgs)

这是当前 demo 最需要补齐的一层。

## 6.3 核心服务层

职责：

- 承载正式 NVR 的稳定业务能力
- 维护核心资源和状态
- 对外暴露稳定服务接口
- 隔离 UI 与平台实现细节

建议拆分为：

- DeviceService
- CapabilityService
- PreviewService
- PlaybackService
- RecordService
- SearchService
- EventService
- StorageService
- ConfigService
- AuthService
- ExportService
- DiagnosticService

## 6.4 平台/HAL 适配层

职责：

- 封装私有 SDK
- 屏蔽不同 SoC 之间的差异
- 提供统一的底层能力抽象
- 统一错误码、状态码、能力表、资源生命周期

建议拆分为：

- Media HAL
- Device HAL
- Storage HAL
- Network HAL
- Display HAL
- AI HAL
- System HAL
- Capability HAL

---

## 7. 核心服务设计

### 7.1 DeviceService

职责：

- 通道/设备资源管理
- 通道状态管理
- 设备信息查询
- 通道能力查询的上层承接

### 7.2 CapabilityService

职责：

- SoC/SDK 能力探测
- 能力表缓存
- 功能开关决策
- 降级策略辅助判断

这是整个项目的关键服务之一。

### 7.3 PreviewService

职责：

- 实时预览会话管理
- 预览资源绑定
- 多画面预览调度
- 码流切换策略

### 7.4 PlaybackService

职责：

- 回放会话管理
- 时间轴控制
- seek / pause / speed 等控制
- 回放状态一致性维护

### 7.5 RecordService

职责：

- 录像启停
- 录像片段管理
- 录像计划承接
- 录像状态维护

### 7.6 SearchService

职责：

- 按时间/事件/通道检索
- 检索条件组织
- 检索结果索引输出

### 7.7 EventService

职责：

- 告警与事件统一管理
- 事件确认/清理
- 告警状态维护
- 后续联动能力挂载点

### 7.8 StorageService

职责：

- 存储介质管理
- 容量/状态管理
- 格式化操作承接
- 导出/空间不足等异常处理

### 7.9 ConfigService

职责：

- 配置加载/保存
- 配置生效流程
- 配置与运行态边界管理

### 7.10 AuthService

职责：

- 登录/登出
- 用户权限与角色管理

### 7.11 ExportService

职责：

- 录像导出
- 日志导出
- 导出进度和失败处理

### 7.12 DiagnosticService

职责：

- 版本、状态、能力、日志、性能等诊断信息统一出口
- 为 SoC 验证和问题定位提供可观察性

## 8. 页面域与组件域建议

### 8.1 页面域

建议至少形成以下稳定页面域：

- 实时预览域
- 回放域
- 检索域
- 事件/告警域
- 设备/通道管理域
- 存储管理域
- 系统配置域
- 用户与安全域
- 诊断与运维域

### 8.2 共性组件域

建议重点沉淀：

- VideoViewport
- ChannelSelector
- TimelinePanel
- SearchFilterPanel
- AlarmList
- ConfigForm
- StatusBar
- ConfirmDialog
- ExportProgressDialog
- CapabilityHint

这些组件不应作为“页面私有零件”存在，而应成为可复用的稳定交互部件。

## 9. 统一资源模型建议

建议优先定义以下核心模型：

- Device
- Channel
- StreamProfile
- CapabilityProfile
- LiveSession
- PlaybackSession
- RecordClip
- SearchQuery
- SearchResult
- AlarmEvent
- StorageVolume
- SystemConfig
- User
- Role
- Permission
- DiagnosticSnapshot

资源模型的意义是：

- 让 UI、服务、HAL 都围绕统一对象协作
- 避免页面自己拼字段
- 避免不同模块对同一对象理解不一致

---

## 10. 状态模型建议

正式系统必须显式定义状态，而不是让状态散落在页面变量和回调里。

建议至少定义以下状态域：

- CurrentPageState
- CurrentChannelState
- PreviewState
- PlaybackState
- TimelineState
- RecordState
- SearchState
- AlarmState
- StorageState
- ConfigState
- CapabilityState
- DiagnosticState
- AuthState

其中关键状态建议如下：

### PreviewState

- idle
- starting
- active
- degraded
- failed
- stopping

### PlaybackState

- idle
- loading
- ready
- playing
- paused
- seeking
- end_of_stream
- failed

### SearchState

- idle
- filtering
- querying
- result_ready
- empty
- failed

### CapabilityState

- detected
- partially_supported
- unsupported
- downgraded
- unknown

能力状态单独建模很重要，因为多 SoC 差异不是异常，而是正式设计输入。

---

## 11. 事件机制设计

建议建立统一事件中心，至少承载以下事件：

- 设备上线/掉线/异常
- 预览启动/停止/失败
- 回放开始/暂停/结束/失败
- 录像启动/停止/空间不足
- 检索完成/为空/失败
- 告警触发/确认/清除
- 配置变更/生效失败
- 存储异常/格式化完成
- 能力探测完成/能力不足
- 导出完成/导出失败
- 性能告警/系统异常

事件机制的作用是：

- 降低散乱回调耦合
- 为状态更新提供统一入口
- 为日志与诊断提供统一链路

---

## 12. 能力探测、裁剪与降级执行

这是本项目区别于普通 NVR UI 工程的关键设计点。

### 12.1 能力探测

系统启动后应完成：

- SoC 型号识别
- SDK 版本识别
- 能力表加载
- 预览/回放/编码/显示/AI/导出等能力识别

### 12.2 功能裁剪

根据能力探测结果，决定：

- 哪些功能显示
- 哪些功能隐藏
- 哪些功能可见但置灰
- 哪些页面行为需要限制

### 12.3 降级执行

建议所有关键动作都支持三种结果语义：

- success
- fail
- degraded

例如：

- 可以预览，但最大同屏路数降低
- 可以回放，但仅支持单路
- 可以导出，但速度受限
- 可以启用某能力，但不能并发运行

降级执行机制必须进入架构主设计，而不是以后再补。

---

## 13. 通道级隔离与资源隔离

多通道系统必须尽量保证：

- 单通道异常不拖垮全局
- 单通道状态可独立诊断
- 单通道资源可单独回收
- 某通道回放失败不影响其他通道预览

这对你们的多 SoC 验证尤其重要，因为资源边界问题常常先在通道级暴露。

---

## 14. 诊断与日志设计

为了服务 SoC 验证和问题定位，建议统一记录：

- 模块名
- 动作名
- 通道号/设备号
- SoC 型号
- SDK 版本
- 错误码
- 状态变化
- 事件 ID / 操作链路 ID
- 是否发生降级

建议 DiagnosticService 统一承接：

- 版本信息
- 能力信息
- 性能统计
- 错误归档
- 日志导出
- 关键链路追踪

如果没有这套诊断设计，后面很难把软件问题和 SoC/SDK 问题区分开。

## 17. 重构与落地优先级

建议按以下顺序推进。

### 第一阶段：抽能力边界

优先抽出：

- CapabilityService
- PreviewService
- PlaybackService
- RecordService
- ConfigService

### 第二阶段：建立统一状态与事件机制

先统一：

- 预览状态
- 回放状态
- 检索状态
- 存储状态
- 能力状态
- 告警状态

### 第三阶段：建立 HAL/Adapter 层

把私有 SDK 与 SoC 差异正式收口。

### 第四阶段：补齐诊断与验证链路

把 SoC 验证能力真正纳入架构，不再依赖临时日志。

### 第五阶段：再推进页面重构

也就是说：

**先重建骨架，再重贴 UI；不要先把 UI 改漂亮，再补底层。**

---

## 18. 风险与边界说明

这份架构文档已经能确定总体方向，但还有几类内容需要在下一阶段继续细化：

- 具体设备接入模型
- 录像文件组织策略
- 检索索引结构
- 用户/权限细粒度设计
- AI 模块接入方式
- 导出/备份的具体流程
- 消息总线实现方式
- 具体 HAL 接口定义

这些不影响当前架构方向成立，但会影响下一阶段详细设计。

---

## 19. 最终结论

当前 NVR 项目不应该继续以 demo 的页面驱动结构扩展，而应该升级为：

**面向多 SoC 兼容、SoC 验证、下游方案支持的 NVR 平台化软件架构。**

这套架构的核心原则是：

- UI 只负责展示与输入
- 动作由控制层翻译为系统用例
- 稳定能力沉淀到服务层
- SoC 差异和 SDK 差异收口到 HAL/Adapter 层
- 统一资源、状态、事件构成正式系统骨架
- 能力探测、功能裁剪、降级执行、诊断能力必须是正式设计内容

这条路线既能保留当前 UI demo 的原型价值，也能避免其结构缺陷继续放大，并且更适合你们当前“平台 + 验证 + 交付”三位一体的真实目标。
