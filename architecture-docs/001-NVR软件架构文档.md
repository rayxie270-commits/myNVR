# NVR 软件架构文档

**版本**：V2.0  
**文档类型**：软件架构设计文档  
**适用阶段**：方案设计、详细设计、开发实施、联调验证、版本演进  
**适用对象**：架构师、开发工程师、测试工程师、平台适配工程师、项目管理人员

---

## 1. 文档目的

本文档定义 NVR 项目的总体软件架构、核心模块划分、层次边界、关键机制与实施约束，用于指导后续开发、联调、版本扩展及平台适配工作。

本文档的目标不是描述某一阶段的调研过程，而是明确项目在工程实现上的统一设计原则，使不同背景成员在加入项目后能够基于同一套架构认知开展工作，降低实现分歧、重复建设与后期维护成本。

---

## 2. 项目定位

本项目定位为一套可持续演进的 NVR 软件平台，而不是单一机型上的页面型演示程序。系统需同时满足以下目标：

1. 支撑正式 NVR 产品能力建设  
2. 支撑多 SoC 平台适配与能力差异收口  
3. 支撑底层多媒体能力验证与问题定位  
4. 支撑后续功能扩展、裁剪交付与版本演进  

因此，软件架构必须具备以下特征：

- 模块边界清晰
- 分层职责稳定
- 能力差异可管理
- 状态与事件可追踪
- UI 与底层实现解耦
- 支持局部重构与持续扩展

---

## 3. 架构设计原则

### 3.1 分层解耦

系统按职责划分为 UI 展示层、交互编排层、核心服务层、平台适配层。各层仅通过受控接口协作，禁止跨层打洞。

### 3.2 业务稳定优先于页面组织

系统主骨架以业务能力、资源对象、状态模型和事件流为核心，不以页面结构作为软件边界依据。

### 3.3 能力差异显式管理

多 SoC、多 SDK 版本之间的能力差异必须通过统一能力模型、能力探测与降级机制进行管理，不允许散落在各页面和业务流程中临时判断。

### 3.4 状态显式化

核心运行状态必须模型化、集中化、可追踪，避免状态分散在页面变量、控件回调、临时标志或底层通知中。

### 3.5 资源生命周期统一管理

预览、回放、录像、存储、导出等关键资源由服务层统一管理，页面不得直接持有和操控底层资源生命周期。

### 3.6 面向实施与演进

架构不仅服务当前版本开发，也必须支持后续功能扩展、平台裁剪、问题定位与新成员接入。

---

## 4. 总体架构

### 4.1 逻辑分层

系统采用四层架构：

1. **UI 展示层**  
2. **交互编排层**  
3. **核心服务层**  
4. **平台适配层（HAL/Adapter）**

### 4.2 总体链路

系统的标准处理链路定义如下：

**用户操作 → UI事件 → 交互编排层 → 核心服务层 → 平台适配层 → 状态/事件回收 → ViewState更新 → UI刷新**

该链路为系统主路径，所有功能实现均应遵循该模式。

```text

│ UI 展示层                                  │
│ Screen / Widget / Dialog / VideoViewport   │
└────────────────────────────────────────────┘
                    │
                    ▼
┌────────────────────────────────────────────┐
│ 交互编排层                                  │
│ PreviewController / PlaybackController     │
│ ConfigController / EventController         │
└────────────────────────────────────────────┘
                    │
                    ▼
┌────────────────────────────────────────────┐
│ 核心服务层                                  │
│ Capability / Device / Preview / Playback   │
│ Record / Search / Event / Storage / Config │
│ Auth / Export / Diagnostic                 │
└────────────────────────────────────────────┘
                    │
                    ▼

                    │
                    ▼

└────────────────────────────────────────────┘
```

## 5. 分层职责定义

## 5.1 UI 展示层

### 职责

- 页面展示
- 控件布局与视觉反馈
- 用户输入采集
- 弹窗、菜单、提示信息呈现
- 消费 ViewState 并刷新界面

### 约束

- 不直接调用 SDK/HAL
- 不直接管理复杂业务状态
- 不直接管理媒体会话与底层资源
- 不承担跨页面业务流程控制

---

## 5.2 交互编排层

### 职责

- 接收 UI 事件
- 将用户动作翻译为系统用例
- 编排跨服务调用流程
- 收敛异步回调和状态变化
- 输出稳定的 ViewState 给 UI

### 典型模块

- PreviewController
- PlaybackController
- SearchController
- RecordController
- ConfigController
- EventController
- StorageController
- DiagnosticController

### 约束

- 不直接调用私有 SDK
- 不直接持有底层资源句柄
- 不承载长期业务规则
- 不代替服务层维护能力模型和系统状态机

---

## 5.3 核心服务层

### 职责

- 提供稳定业务能力
- 维护领域状态与资源生命周期
- 统一承接策略判断
- 管理能力约束、异常处理与降级逻辑
- 对上提供稳定服务接口，对下调用平台适配层

### 核心服务

- DeviceService
- CapabilityService
- PreviewService
- PlaybackService
- RecordService
- SearchService
- EventService
- StorageService
- ConfigService
- AuthService
- ExportService
- DiagnosticService

---

## 5.4 平台适配层

### 职责

- 封装底层私有 SDK
- 收口不同 SoC 的差异
- 提供统一底层接口语义
- 统一错误码、状态码与能力查询结果
- 为服务层屏蔽平台实现细节

### 典型模块

- Media HAL
- Display HAL
- Storage HAL
- Device HAL
- Network HAL
- System HAL
- Capability HAL
- SoC Adapter

### 约束

- 不感知页面语义
- 不返回 UI 展示文本
- 不参与页面流程编排
- 不向上泄露平台私有细节

---

## 6. 核心能力域划分

系统按业务能力划分为以下域：

1. 实时预览域  
2. 回放与录像域  
3. 检索与索引域  
4. 事件与告警域  
5. 设备与通道管理域  
6. 存储管理域  
7. 系统配置域  
8. 用户与权限域  
9. 诊断与运维域  

这些能力域是系统设计边界，不等同于页面划分。

---

## 7. 核心模型

建议统一建立以下核心模型：

- Device
- Channel
- StreamProfile
- CapabilityProfile
- LiveSession
- PlaybackSession
- RecordTask
- SearchQuery
- SearchResult
- AlarmEvent
- StorageVolume
- SystemConfig
- User
- Permission
- DiagnosticSnapshot

统一模型的目标是确保不同模块围绕同一对象语义协作，避免字段拼接和理解分裂。

## 9. 事件机制

系统建立统一事件中心，承载系统运行中的状态广播与异步通知。事件类型至少包括：

- DeviceEvent
- PreviewEvent
- PlaybackEvent
- RecordEvent
- SearchEvent
- AlarmEvent
- StorageEvent
- ConfigEvent
- CapabilityEvent
- DiagnosticEvent

### 事件约束

- 事件用于广播状态变化，不用于绕开服务边界
- 关键事件必须携带来源模块、对象标识、时间戳、traceId
- 故障、超时、降级、恢复等关键变化必须事件化

---

## 10. 能力探测、裁剪与降级

### 10.1 能力探测

系统启动时完成：

1. SoC 识别  
2. SDK 版本识别  
3. HAL 能力查询  
4. 能力归一化  
5. 初始功能策略生成  

### 10.2 功能裁剪

根据能力探测结果，决定：

- 功能是否显示
- 功能是否可交互
- 功能是否需受限运行
- 页面是否启用某些高级能力

### 10.3 降级机制

关键业务接口统一支持以下结果语义：

- success
- fail
- degraded

例如：

- 预览可用，但同屏路数下降
- 回放可用，但只支持单路
- 导出可用，但性能受限
- 某能力可启用，但并发受限

降级是正式设计的一部分，不是补丁策略。

## 11. 关键模块设计

## 11.1 CapabilityService

职责：

- 统一能力探测
- 能力缓存与查询
- 为功能裁剪和降级提供决策输入

约束：

- 上层不得各自绕过它独立判断 SoC 能力

---

## 11.2 PreviewService

职责：

- 管理预览会话
- 处理画面绑定、码流选择、布局调度
- 维护预览生命周期与异常恢复

约束：

- 预览会话生命周期由服务层统一管理
- 单通道故障不应拖垮全局

---

## 11.3 PlaybackService

职责：

- 管理回放会话
- 承接时间轴控制、seek、pause、speed切换
- 维护回放状态一致性

---

## 11.4 ConfigService

职责：

- 配置读取、校验、保存、生效协调
- 区分保存成功与生效成功
- 管理配置变更传播

---

## 11.5 StorageService

职责：

- 管理存储设备状态、容量、格式化、异常
- 提供导出与空间管理支撑
- 协调存储异常时的保护动作

---

## 11.6 DiagnosticService

职责：

- 统一日志、追踪、错误上下文和运行快照
- 支撑 SoC 验证和问题定位
- 提供导出型诊断材料

---

## 12. 层间接口约束

### 12.1 允许的调用方向

- UI → Controller
- Controller → Service
- Service → HAL

### 12.2 禁止的调用方向

- UI → HAL
- UI → SDK
- Controller → SDK
- HAL → UI
- 页面直接跨多个服务拼流程

### 12.3 统一结果语义

服务接口不得仅返回布尔值，需统一使用结构化结果，至少表达：

- 成功
- 失败
- 降级成功

### 12.4 统一错误分类

错误至少分为：

- 用户输入错误
- 配置错误
- 能力不足
- 资源不足
- 状态冲突
- 超时
- HAL/SDK错误
- 系统异常

---

## 13. 关键流程

## 13.1 实时预览流程

1. UI 发起预览动作  
2. PreviewController 解析请求并检查页面上下文  
3. CapabilityService 返回能力约束  
4. PreviewService 申请资源并选择执行策略  
5. Media HAL / Display HAL 执行底层操作  
6. PreviewService 更新预览状态  
7. Controller 映射 ViewState  
8. UI 刷新画面与交互状态  

---

## 13.2 回放流程

1. UI 发起检索  
2. SearchController 调用 SearchService  
3. 用户选择结果  
4. PlaybackController 组装回放请求  
5. PlaybackService 校验条件并创建回放会话  
6. HAL 执行底层回放  
7. 回放状态和时间轴同步到 UI  

---

## 13.3 配置保存与生效流程

1. UI 提交配置  
2. ConfigController 做交互层校验  
3. ConfigService 做业务校验与持久化  
4. ConfigService 协调生效动作  
5. 相关服务接收配置变更  
6. UI 根据最终状态显示结果  

---

## 14. 通道隔离与异常处理

系统必须支持通道级隔离，至少满足：

- 单通道故障不拖垮其他通道
- 单通道资源可独立释放
- 单通道状态可独立诊断
- 故障与恢复过程可追踪

该机制对多通道预览、回放和录像尤为关键。

---

## 15. 日志与诊断要求

关键链路必须具备统一追踪能力，日志中至少包含：

- 模块名
- 动作名
- 通道号/设备号
- SoC 型号
- SDK 版本
- 错误码
- 状态变化
- traceId / actionId
- success / fail / degraded 标记

日志系统必须支持问题回溯、版本对比和导出分析。

## 17. 实施路线

### 第一阶段：建立主骨架

- 落地 Controller / Service / HAL 分层
- 抽取 Preview、Playback、Config、Capability 四条主链路

### 第二阶段：统一状态与事件

- 落地核心状态机
- 建立统一事件中心和结果语义

### 第三阶段：完成平台收口

- 将 SDK 调用下沉到 HAL/Adapter
- 建立 SoC 差异隔离层

### 第四阶段：补齐诊断与验证能力

- 接入 traceId、关键快照与统一错误链路
- 为 SoC 验证和联调提供可观察性

### 第五阶段：推进页面与组件治理

- 在骨架稳定后再做页面整理和共性组件沉淀
- 避免先改 UI、后补基础架构

---

## 18. 架构约束总结

本项目开发过程中必须遵守以下约束：

1. 页面不得直接调用 HAL/SDK  
2. 业务流程不得散落在多个页面中拼装  
3. 能力探测、裁剪、降级必须纳入正式设计  
4. 状态必须统一建模，禁止页面私有状态泛滥  
5. 核心服务接口必须稳定、结构化、可追踪  
6. SoC 差异必须收口到平台适配层  
7. 新增功能必须优先明确能力域、状态和接口边界，再进入实现  

---

## 19. 结论

本项目的软件架构目标，是建设一套面向多 SoC 兼容、正式产品交付、问题定位和持续演进的 NVR 软件平台。系统必须从“页面驱动型工程”升级为“分层明确、状态可控、能力可管理、平台可扩展”的正式软件架构。

后续所有开发工作，均应以本文档定义的分层边界、核心模型、状态机制和实施约束为基准。
