# 003《本地 UI 代码架构审查》

## 1. 结论

当前这套 NVR UI demo，**可以继续作为外观与交互参考，但不适合直接演进成正式产品的软件骨架**。

核心原因不是“代码一定写得差”，而是它天然更像一个**页面驱动的演示工程**，而你们当前项目的真实目标已经明显升级为：

- 适配多颗不同性能的 SoC
- 运行在 Linux 4.9 / 5.10
- 依赖私有多媒体 SDK（但总体参考海思链路）
- 同时服务于 SoC 验证、能力展示、下游方案支持
- 后续项目结果还会反向影响下一代 SoC 设计

在这个前提下，正式 NVR 项目必须承担的不只是“把页面做出来”，而是：

- 稳定承接不同 SoC 的能力差异
- 为底层媒体链路与系统能力提供统一抽象
- 为后续 HAL 接口、自动化验证、功能裁剪、多机型适配留出结构空间
- 避免 UI demo 结构反过来绑死平台演进

所以我的判断很明确：

**当前 UI demo 的价值主要在“界面原型、交互入口、能力暴露范围”，而不在“软件架构本体”。**

正式版本建议转向：

**页面控制器 + 应用服务 + 领域服务 + 平台/HAL 适配** 的分层架构。

---

## 2. 当前项目所处的特殊背景

这部分很关键，因为它决定了审查口径不能照普通消费级 NVR 或单一机型项目来做。

### 2.1 项目不是单纯终端应用，而是 SoC 能力承载平台

你们的 NVR 项目本质上同时承担三层任务：

1. **应用交付层**
   - 给最终下游客户一个可运行的 NVR 产品基础
2. **SoC 能力验证层**
   - 验证视频编解码、显示、存储、网络、AI、稳定性等能力
3. **SoC 设计反馈层**
   - 当前项目暴露出来的问题，后续可能成为下一代 SoC 架构/SDK 设计输入

这意味着：

- 软件架构不能只为当前 UI 服务
- 必须能把底层能力问题、性能问题、接口缺陷暴露出来
- 必须具备较强的诊断性、裁剪性、可迁移性

### 2.2 多 SoC 兼容是核心约束，不是附加需求

你们不是一套软件只跑一颗芯片，而是要兼容多颗性能不同、能力不同的 SoC。

这直接要求：

- 设备能力必须抽象
- 媒体通路必须抽象
- UI 不能写死依赖某一套链路细节
- 功能开关和能力探测必须成为正式机制

### 2.3 私有 SDK + 海思参考路径意味着“可借鉴，但不能硬绑定”

虽然你们多媒体链路参考海思平台，但因为：

- 命名可能不同
- 实现细节可能不同
- 不同 SoC 的能力边界不一致
- 后续 SDK 还会演进

所以应用层绝不能把“某平台 SDK 习惯”硬编码成产品架构事实。

## 3. 当前 UI demo 架构的主要问题判断

以下问题不是指某个文件一定错，而是从正式产品演进角度看，当前 demo 结构大概率会暴露出的通病。

### 3.1 页面驱动业务，而不是业务驱动页面

这是我认为最核心的问题。

如果页面是系统主线，通常会出现：

- 预览页面里塞预览逻辑
- 回放页面里塞回放逻辑
- 设置页面里塞配置读写逻辑
- 告警页面里塞告警处理逻辑

短期看很直观，长期会变成：

- 同一业务在多个页面重复实现
- 状态来源不唯一
- 页面切换导致业务状态丢失或难跟踪
- 新增机型/能力时需要到处改 UI

对你们这种项目，这会非常危险，因为底层能力本来就在变化。

### 3.2 UI 与媒体控制链路过近

NVR 里最容易失控的不是普通配置页，而是：

- 实时预览
- 回放
- 录像
- 解码/显示
- 抓图
- 音频对讲
- AI 相关触发

如果 UI 直接控制这些链路，常见后果是：

- 页面生命周期和媒体生命周期混在一起
- 一个页面退出后资源释放不干净
- 多窗口/多通道扩展时难以复用
- 切换码流、切换分辨率、切换 SoC 时问题爆炸

### 3.3 状态管理容易散落在控件、页面和回调里

demo 工程经常有这个特征：

- 某个按钮自己记状态
- 页面类自己记一部分状态
- 异步回调又更新另一部分状态
- 定时器或消息回调再改一次显示

结果是：

- 状态来源分裂
- UI 刷新时机不可控
- 异常恢复路径不清楚
- 调试困难

而 NVR 这种多异步源系统，本身就会不断收到：

- 设备上下线
- 录像状态变化
- 存储变化
- 回放结束
- 网络异常
- AI 事件

如果没有统一状态模型，复杂度会快速失控。

### 3.4 控件事件与业务动作缺少明确翻译层

正式系统里，一个按钮点击不应直接等于一串底层调用。

更合理的方式是：

- UI 只表达用户意图
- 由控制器/用例层翻译成系统动作
- 再调用服务层
- 服务层再访问 HAL/SDK

否则会出现：

- 同一个按钮逻辑难复用
- 很难写自动化测试
- UI 改动牵扯底层
- 底层能力调整又反向污染页面代码

### 3.5 缺少“能力探测 → 功能裁剪 → UI 呈现”闭环

这是多 SoC 项目特别关键的一点。

正式产品不能假设所有 SoC 都支持相同能力。

例如不同 SoC 可能在这些方面存在差异：

- 最大接入路数
- 同屏预览能力
- 回放并发能力
- AI 能力有无
- 编码格式支持范围
- 解码性能
- 显示输出规格
- 抓图/转码/导出能力

如果 UI 结构没有能力探测和裁剪机制，就会变成：

- 页面上出现实际不可用按钮
- 某些功能点运行到中途才失败
- 机型间分支到处开花

### 3.6 诊断能力可能不足

由于你们项目还承担 SoC 验证角色，所以系统不仅要“能跑”，还要能回答：

- 哪个模块慢
- 哪条媒体链路失败
- 哪个 SoC 上退化明显
- 某能力缺失是 UI 问题、服务问题还是 SDK 问题

如果当前 demo 的日志、状态、错误码、事件链都不统一，后面排查会越来越困难。

## 4. 面向正式产品的风险清单

### 4.1 可维护性风险

如果继续以页面驱动方式扩展，后面会出现：

- 一个需求改多处页面
- 逻辑重复越来越多
- 新人很难定位某个能力到底归谁管

### 4.2 可扩展性风险

未来增加这些能力时，现有结构很可能吃力：

- 更多 SoC 型号
- 更多通道数
- 不同编解码策略
- AI 检测/事件联动
- 导出/备份/远程查看
- 多套 UI 皮肤或定制版本

### 4.3 稳定性风险

媒体能力一旦和页面耦合，常见问题包括：

- 切页面黑屏
- 解码句柄泄漏
- 回放结束状态错乱
- 异常恢复不完整
- 某通道错误影响全局

### 4.4 可测试性风险

如果没有中间的用例层/服务层，测试会被迫依赖 UI 驱动，导致：

- 自动化测试难写
- 问题难复现
- SoC 对比验证缺少统一测试入口

### 4.5 平台演进风险

这是最要命的一点：

如果项目结果还要反馈给下一代 SoC 设计，那么当前架构必须能清晰分离：

- SoC 本身能力边界
- SDK 封装问题
- 产品逻辑问题
- UI 展现问题

否则最后得不出有价值的设计反馈。

## 5. 推荐的分层重构方向

我建议正式架构至少收敛成四层。

### 5.1 UI 展示层

职责：

- 页面显示
- 控件交互
- 菜单/弹窗/对话框
- 主题、布局、焦点逻辑

原则：

- 不直接碰 SDK
- 不直接管理复杂媒体资源
- 不直接维护跨页面业务状态

### 5.2 页面控制与应用编排层

这是当前 demo 最应该补出来的一层。

职责：

- 把“按钮点击/页面进入”翻译成系统用例
- 管理页面状态与异步流程
- 组织跨服务流程
- 对 UI 输出稳定的 ViewModel / 页面状态

典型用例：

- 打开通道预览
- 切换子码流
- 按条件检索录像
- 开始回放并跳转时间轴
- 确认告警
- 保存配置并触发生效

### 5.3 领域服务层

职责：

- 承载正式 NVR 的业务能力
- 统一状态与规则
- 为 UI/控制层提供稳定接口

建议至少拆出这些服务：

- DeviceService
- CapabilityService
- PreviewService
- PlaybackService
- RecordService
- SearchService
- EventService
- StorageService
- ConfigService
- User/AuthService
- DiagnosticService

其中 **CapabilityService** 很重要，用于承接不同 SoC/不同版本 SDK 的能力差异。

### 5.4 平台/HAL 适配层

职责：

- 封装私有 SDK
- 适配不同 SoC 的差异
- 提供统一错误码/状态/能力查询
- 承接 OS、驱动、媒体链路、显示、存储、网络等底层接口

建议这一层明确区分：

- Media HAL
- Display HAL
- Storage HAL
- Network HAL
- AI HAL
- System HAL

## 6. 推荐的关键机制

### 6.1 能力探测与功能裁剪机制

系统启动后应先完成：

- SoC 型号识别
- SDK 版本识别
- 能力表加载
- 功能开关决策
- UI 功能裁剪

这样才能保证：

- 同一套 UI 可适配多 SoC
- 不同机型行为可预测
- 问题更容易定位

### 6.2 统一事件中心

建议统一承载：

- 设备状态变化
- 媒体状态变化
- 录像状态变化
- 存储异常
- 告警/AI 事件
- 用户配置变更
- 系统诊断事件

事件中心的价值不是“看起来先进”，而是减少散乱回调耦合。

### 6.3 统一状态模型

建议至少明确这些状态域：

- DeviceState
- PreviewState
- PlaybackState
- RecordState
- StorageState
- AlarmState
- PageState
- CapabilityState

### 6.4 通道级隔离

对于多通道场景，建议尽量保证：

- 单通道异常不拖垮全局
- 单通道状态可独立诊断
- 单通道资源可单独回收

### 6.5 诊断与日志统一出口

建议把以下信息统一化：

- 模块名
- 通道号
- 设备号
- SoC 型号
- SDK 版本
- 错误码
- 操作链路 ID

否则后续对比不同 SoC 的问题时，几乎无法高效追踪。

## 7. 建议的代码组织方向

下面给一个更贴近落地的代码组织建议，不绑定具体语言和 GUI 框架。

```text
app/
  ui/
    screens/
    widgets/
    dialogs/
    presenters/
  controller/
    preview/
    playback/
    record/
    config/
    event/
  service/
    device/
    capability/
    preview/
    playback/
    record/
    search/
    event/
    storage/
    config/
    diagnostic/
  domain/
    model/
    state/
    event/
    errors/
  platform/
    hal/
      media/
      display/
      storage/
      network/
      ai/
      system/
    adapter/
      hisi_like/
      soc_x/
      soc_y/
  infra/
    log/
    config/
    message_bus/
    persistence/
```

重点不是目录长什么样，而是：

- UI 不跨层直接打到 HAL
- SoC 差异收口到 platform/adapter
- 统一状态和事件模型放到 domain
- 诊断能力不是散落在每个页面里

## 8. 重构优先级建议

如果你们要从当前 demo 向正式平台迁移，我建议优先级这样排。

### 第一优先级：抽出能力边界

先做：

- PreviewService
- PlaybackService
- RecordService
- ConfigService
- CapabilityService

目的：

把最核心、最容易耦合 UI 的能力先收口。

### 第二优先级：建立统一状态与事件机制

先把：

- 页面状态
- 设备状态
- 媒体状态
- 存储状态
- 告警状态

统一起来。

### 第三优先级：建立 HAL/SDK 适配层

把不同 SoC / SDK 差异从业务层剥离出来。

### 第四优先级：补诊断链路

为了服务 SoC 验证，这一块不能省。

### 第五优先级：再做页面层整理

也就是说：

**不要先大改 UI，再补底层；应该先把业务骨架抽出来，再让 UI 贴上去。**

## 9. 对 001 主文档的输入价值

003 这份审查，最终给 001 主文档提供这些关键输入：

1. 当前 demo 更适合作为原型，不适合作为正式平台骨架
2. 正式架构必须显式支持多 SoC 能力差异
3. 能力探测、功能裁剪、统一状态、统一事件中心是刚需
4. 媒体链路与 UI 必须解耦
5. 项目必须兼顾产品交付与 SoC 验证双重目标
6. 后续 HAL 文档应围绕能力边界来抽，而不是围绕页面按钮来抽

---

## 10. 最终建议

一句话概括：

**当前 UI demo 可以保留“长相”，但必须重建“骨架”。**

更具体一点：

- UI 保留现有页面原型价值
- 业务能力抽成稳定服务
- SoC 差异压到 HAL/adapter
- 状态、事件、诊断做成统一机制
- 把项目从“演示工程”推进成“平台化 NVR 基座”

这条路线既适合你们当前多 SoC 兼容，也更符合“为 SoC 开发测试服务、为下游提供整套支持”的真实目标。
