# 004《按钮关系文档结构化整理》

## 1. 结论

当前这份“按钮与父子关系”类输入，如果只把它当作 UI 清单来用，价值会比较有限；但如果把它进一步结构化成**页面树、组件树、动作树、状态树、服务映射树**，它就能直接成为后续软件架构设计和 HAL 接口抽取的桥梁。

对你们这个 NVR 项目来说，004 的真正价值不是“罗列有哪些按钮”，而是把这些按钮背后的东西抽出来：

- 页面如何组织
- 功能入口如何分层
- 哪些是全局动作，哪些是局部动作
- 哪些动作只是 UI 行为，哪些动作其实对应系统用例
- 哪些入口会落到设备、媒体、存储、告警、配置、诊断等服务层

所以这份整理的核心目标应该定为：

**把 UI 交互入口，翻译成未来正式 NVR 平台的功能结构图谱。**

---

## 2. 为什么 004 很关键

003 更多是在讲“当前代码结构不该怎么继续长”；
004 则是在解决另一个问题：

**如果不按当前 demo 的代码结构走，那现有页面和按钮信息还能怎么被保留下来？**

答案就是：

- 不保留其代码耦合方式
- 保留其交互入口与业务意图
- 再重新映射到正式的软件架构边界中

这一步如果做得好，后面就能形成：

- UI 页面设计依据
- 应用服务列表
- HAL 接口分类基础
- 状态机梳理基础
- 自动化测试用例入口清单

---

## 3. 建议的结构化方式

我建议不要只做“页面 → 按钮”两层树，而是至少做五层结构。

### 3.1 页面树（Page Tree）

描述系统有哪些一级页面、二级页面、弹窗、菜单、对话框。

例如：

- 首页 / 预览页
- 回放页
- 录像检索页
- 事件/告警页
- 设置页
  - 通道设置
  - 录像设置
  - 网络设置
  - 存储设置
  - 系统设置
- 弹窗
  - 时间选择
  - 确认框
  - 通道选择框
  - 导出进度框

页面树的作用是：

- 定义导航结构
- 定义 UI 层次
- 定义哪些功能属于常驻页面，哪些只是临时交互

### 3.2 组件树（Component Tree）

页面内部还要继续拆：

- 页面级区域
- 面板
- 工具栏
- 视频窗口
- 时间轴
- 列表区
- 详情区
- 状态栏
- 菜单栏

这一步的价值在于区分：

- 某个按钮是页面级动作
- 还是某个组件私有动作
- 还是全局共用控件

### 3.3 动作树（Action Tree）

这是最关键的一层。

按钮不应只记录“名字”，而应该记录它是什么类型的动作。

建议分成：

- 导航动作
  - 切页
  - 返回
  - 打开弹窗
- 视图动作
  - 切换布局
  - 排序
  - 筛选
  - 缩放
- 媒体动作
  - 开始预览
  - 停止预览
  - 开始回放
  - 暂停
  - 快进
  - 抓图
  - 开启录像
- 资源动作
  - 选择通道
  - 选择磁盘
  - 选择时间段
- 配置动作
  - 修改参数
  - 应用
  - 保存
  - 恢复默认
- 事件动作
  - 确认告警
  - 清除提示
  - 查看详情
- 系统动作
  - 登录
  - 注销
  - 重启
  - 升级
  - 导出日志

这一步一做，UI 就不再只是 UI，而是开始变成系统用例清单。

### 3.4 状态树（State Tree）

每个动作都应该问一句：

**它影响什么状态？**

例如：

- 开始预览 → PreviewState
- 切换通道 → CurrentChannelState + PreviewState
- 开始回放 → PlaybackState + TimelineState
- 导出录像 → ExportTaskState
- 保存配置 → ConfigState + DirtyState
- 确认告警 → AlarmState

如果没有状态树，后面就会出现：

- 页面看起来能点
- 但不知道哪些地方应该刷新
- 不知道失败后该恢复哪个状态

### 3.5 服务映射树（Service Mapping Tree）

最终每个动作都应该找到它对应的目标服务。

例如：

- 打开预览 → PreviewService
- 切换码流 → PreviewService + CapabilityService
- 回放检索 → SearchService
- 回放控制 → PlaybackService
- 导出录像 → ExportService / RecordService
- 磁盘格式化 → StorageService
- 保存网络参数 → ConfigService + NetworkService
- 查看设备信息 → DeviceService + DiagnosticService

这一步会直接变成后续 HAL 接口和模块边界的输入。

## 4. 建议形成的文档产物

我建议 004 最终不要只是一篇描述文，而是至少包含下面几类结构化结果。

### 4.1 页面功能总表

字段建议：

- 页面名
- 父页面
- 页面类型（常驻页/弹窗/菜单/对话框）
- 主要功能
- 依赖服务
- 备注

### 4.2 按钮/入口动作总表

字段建议：

- 所属页面
- 所属组件
- 入口名称
- 动作类型
- 触发条件
- 影响状态
- 调用服务
- 是否依赖 SoC 能力
- 是否需要 HAL 支持

### 4.3 全局入口清单

比如：

- 返回
- 首页
- 登录/登出
- 通道切换
- 布局切换
- 时间选择
- 搜索
- 告警提示入口
- 系统状态入口

这类入口通常会跨页面复用，应该优先抽成共性能力。

### 4.4 共性组件清单

比如：

- 通道选择组件
- 日期时间组件
- 视频窗口组件
- 时间轴组件
- 操作确认框
- 状态提示组件
- 列表筛选组件
- 导出进度组件

这些组件最好不要每个页面各做一套。

### 4.5 服务映射清单

也就是把页面动作聚类后，归到这些服务名下：

- DeviceService
- PreviewService
- PlaybackService
- RecordService
- SearchService
- EventService
- StorageService
- ConfigService
- User/AuthService
- DiagnosticService
- CapabilityService

## 5. 从 004 能抽出的关键结构结论

### 5.1 页面不是系统主线，动作和状态才是

只看页面树，会误以为系统是“页面集合”；
但从按钮关系继续抽象后会发现，真正稳定的其实是：

- 动作类型
- 状态变化
- 服务边界

这也印证了 003 的结论：

**正式软件架构不该围绕页面组织。**

### 5.2 很多按钮其实只是同一能力的不同入口

比如这些动作看起来分散在不同页面：

- 预览页点击窗口开始播放
- 通道列表选择某路视频
- 双击窗口放大
- 回放页选择通道

本质上都可能落到：

- 通道选择
- 预览会话建立
- 视图状态变更

也就是说，UI 入口很多，但底层服务能力未必很多。

### 5.3 一部分按钮是“视图动作”，不该污染业务服务

例如：

- 翻页
- 排序
- 切布局
- 展开/收起
- 聚焦切换

这类动作应主要停留在 UI / Controller 层，不能和底层能力混在一起。

### 5.4 一部分按钮是“真正业务动作”，必须进入服务层

例如：

- 开始录像
- 停止录像
- 回放检索
- 确认告警
- 存储格式化
- 导出录像
- 保存配置

这些才是需要进入正式平台骨架的动作。

### 5.5 有些按钮其实是在暴露 SoC 能力边界

对你们来说，这一点特别重要。

比如这些动作背后不只是 UI：

- 多画面预览
- 高分辨率解码
- 多路同步回放
- 导出/转码
- AI 检测开关
- 音频对讲
- 抓图

这些其实是在反映：

- SoC 算力上限
- 媒体链路能力
- SDK 完整度
- 存储/网络吞吐能力

所以 004 不只是给 UI 用，也是在帮助你们识别 SoC 验证点。

## 7. 004 对 HAL 接口设计的意义

004 的另一个关键价值，是它能为后续 HAL 文档提供非常自然的接口分类基础。

因为很多按钮最后都能反推到底层能力类别：

### 7.1 媒体类 HAL

对应动作：

- 预览
- 回放
- 暂停
- 快进/快退
- 抓图
- 录像
- 导出

### 7.2 设备类 HAL

对应动作：

- 通道启停
- 设备状态查询
- 参数读取
- 云台控制
- 音频开关

### 7.3 存储类 HAL

对应动作：

- 磁盘查询
- 容量显示
- 格式化
- 录像目录管理
- 导出目标管理

### 7.4 系统类 HAL

对应动作：

- 时间同步
- 网络参数
- 系统升级
- 重启
- 恢复默认
- 日志导出

### 7.5 能力类 HAL

对应动作：

- 查询支持的预览路数
- 查询支持的回放并发数
- 查询是否支持 AI
- 查询编码/解码规格
- 查询显示输出规格

这意味着：

**按钮关系文档不是 UI 附属品，而是后续接口设计的重要输入。**

## 8. 建议的落地流程

我建议你们后续对 004 这样落地。

### 第一步：把原始按钮清单清洗成结构化表格

至少包含：

- 页面
- 父组件
- 入口名称
- 动作描述

### 第二步：给每个入口打标签

标签建议包括：

- 导航类
- 视图类
- 业务类
- 系统类
- 诊断类
- SoC 能力相关

### 第三步：补状态影响字段

至少写清楚：

- 影响哪个状态域
- 是否异步
- 是否可能失败
- 失败后 UI 如何反馈

### 第四步：补服务映射字段

把每个动作归入目标服务。

### 第五步：从表格反推出模块边界和 HAL 分类

这一步完成后，004 就不只是“整理按钮”，而是已经在为 001 和 HAL 文档服务。

## 9. 对 001 主文档的输入价值

004 最终会给 001 提供这些非常直接的输入：

1. 页面导航结构
2. 共性组件结构
3. 全局动作入口清单
4. 系统用例清单
5. 状态域拆分基础
6. 服务边界候选清单
7. HAL 接口分类基础
8. SoC 能力验证点清单

也就是说，004 是把“看得见的 UI”翻译成“能落地的软件架构输入”。

---

## 10. 最终建议

一句话概括：

**不要把按钮关系文档当成 UI 附件，要把它当成系统动作模型的原材料。**

更具体一点：

- 页面树保留导航信息
- 组件树保留复用边界
- 动作树抽出系统用例
- 状态树定义刷新与恢复逻辑
- 服务映射树反推模块和 HAL

这样整理完之后，004 才真正能支撑后续正式 NVR 架构设计，而不是停留在“按钮列表汇总”这个低价值层面。
