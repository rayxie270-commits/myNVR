# 004《按钮关系文档结构化整理（架构支撑版）》

## 1. 结论

这份文档不再把“按钮关系”当成简单的 UI 清单，而是把它正式升级为：

**面向 NVR 软件架构设计的交互结构化输入。**

也就是说，004 的目标不是回答“有哪些按钮”，而是回答下面这些对架构真正有价值的问题：

- 系统有哪些稳定页面域
- 页面内部有哪些稳定组件域
- 用户动作里哪些是视图动作，哪些是业务动作，哪些是系统动作
- 这些动作分别落到哪些服务边界
- 哪些动作会影响状态机
- 哪些动作需要 HAL/SDK 能力支撑
- 哪些动作天然就是 SoC 验证点

对当前项目来说，这一步尤其关键。因为你们的 NVR 项目不是普通消费级单机应用，而是同时承担：

- 多 SoC 兼容
- 私有多媒体 SDK 承接
- SoC 能力验证
- 下游方案支持
- 后续 SoC 设计反馈输入

所以按钮关系文档如果不被结构化，最后只会停留在“页面梳理”；
而一旦结构化到位，它就能直接支撑：

- 001《NVR 软件架构文档》
- 后续 HAL 接口文档
- 页面控制器/应用服务设计
- 状态模型设计
- 自动化测试入口设计

---

## 2. 004 的定位

### 2.1 它不是按钮清单，而是动作模型输入

按钮只是表象，真正需要沉淀的是：

- 动作类型
- 状态变化
- 服务落点
- 能力依赖
- 页面与组件边界

因此 004 的最终使用对象，不只是 UI 开发，而是：

- 架构设计
- 应用层设计
- HAL 设计
- 测试设计
- SoC 验证设计

### 2.2 它是 003 和 001 之间的桥

- 003 解决的是：当前 demo 架构为什么不能直接长大
- 004 解决的是：现有 UI 信息怎样被重新翻译成正式架构输入
- 001 则会在 003/004 之上统一输出正式 NVR 架构

所以 004 的任务不是描述 UI，而是：

**把 UI 交互入口翻译成正式平台的动作、状态、服务、接口语言。**

## 3. 结构化产出目标

为了让 004 真正能支撑架构设计，我建议最终固定输出下面五类结构。

### 3.1 页面域（Page Domains）

系统不应该只是一串页面，而应该划分为稳定的页面域。

建议当前 NVR 至少抽象成下面这些页面域：

1. **实时预览域**
2. **录像回放域**
3. **检索筛选域**
4. **事件/告警域**
5. **设备/通道管理域**
6. **存储管理域**
7. **系统配置域**
8. **用户与安全域**
9. **诊断与运维域**

这一步的意义是：

- 页面不再是零散入口
- 页面开始对应业务域
- 后续服务边界能和页面域形成映射，但不被页面绑死

### 3.2 组件域（Component Domains）

页面内部要抽出跨页面稳定复用的组件域。

建议重点沉淀这些共性组件：

- VideoViewport（视频窗口）
- ChannelSelector（通道选择器）
- TimelinePanel（时间轴）
- SearchFilterPanel（筛选面板）
- AlarmList / EventList（事件列表）
- ConfigForm（配置表单）
- StatusBar（状态栏）
- ConfirmDialog（确认框）
- ExportProgressDialog（导出进度框）
- CapabilityHint（能力提示组件）

这一步的意义是：

- 识别共性复用单元
- 避免每个页面各自实现相似逻辑
- 为后续 Presenter/ViewModel 或 Controller 边界做准备

### 3.3 动作域（Action Domains）

按钮背后真正稳定的是动作，不是名字。

建议把动作统一归类为：

1. **导航动作**
   - 页面切换、返回、打开弹窗、关闭弹窗
2. **视图动作**
   - 布局切换、排序、缩放、分页、展开/收起
3. **资源选择动作**
   - 选择通道、选择设备、选择时间段、选择磁盘
4. **媒体控制动作**
   - 开始预览、停止预览、切码流、开始回放、暂停、拖动时间轴、抓图、导出
5. **记录与检索动作**
   - 条件检索、录像查询、事件查询、结果过滤
6. **事件处理动作**
   - 查看告警、确认告警、清除提示、查看详情
7. **配置动作**
   - 修改配置、应用配置、保存、恢复默认
8. **系统动作**
   - 登录、登出、升级、重启、恢复出厂、导出日志
9. **诊断动作**
   - 查看状态、查看版本、查看性能、查看能力表

这一步的意义是：

- 把 UI 入口翻译成系统用例
- 后续服务设计有了稳定输入
- 自动化测试也可以围绕动作域建模

### 3.4 状态域（State Domains）

动作一旦发生，就会影响状态。

建议至少固定这些状态域：

- CurrentPageState
- CurrentChannelState
- PreviewState
- PlaybackState
- TimelineState
- RecordState
- SearchState
- AlarmState
- StorageState
- ConfigState
- CapabilityState
- DiagnosticState
- AuthState

这一步的意义是：

- 为 UI 刷新建立依据
- 为异常恢复建立依据
- 为多 SoC 差异处理建立依据

### 3.5 服务域（Service Domains）

最终所有有效动作都要落到服务边界。

建议当前项目按下面这些服务域组织：

- DeviceService
- CapabilityService
- PreviewService
- PlaybackService
- RecordService
- SearchService
- EventService
- StorageService
- ConfigService
- AuthService
- ExportService
- DiagnosticService

这一步的意义是：

- 让 UI 入口和正式平台骨架连接起来
- 为后续 HAL 接口分类提供上层语义

不仅是画面显示窗口，它还承担：

- 预览状态呈现

## 7. 状态域设计建议

为了让 004 能真正支撑架构，而不是停留在说明层，这里把关键状态域进一步落地。

- idle

- active

- degraded
- failed
- stopping

### 7.2 PlaybackState

建议至少包含：

- idle
- loading
- ready
- playing
- paused
- seeking
- end_of_stream
- failed

### 7.3 SearchState

建议至少包含：

- idle
- filtering
- querying
- result_ready
- empty
- failed

### 7.4 StorageState

建议至少包含：

- normal
- warning
- full
- unavailable
- formatting
- failed

### 7.5 CapabilityState

建议至少包含：

- detected

这几个状态域有两个直接收益：

1. UI 层不再自己乱拼状态

## 8. 004 对多 SoC 项目的特殊价值

这一部分是这次整理后必须明确写进去的，因为它决定 004 不只是 UI 文档。

### 8.1 按钮不是按钮，它们是能力探针

对普通应用来说，一个按钮只是一个功能入口；
但对你们来说，很多按钮其实是在问底层：

- 支不支持
- 能跑几路
- 是否会退化
- 失败如何反馈
- 哪个 SoC 上表现不同

因此像下面这些动作，都应该被视为 SoC 验证点：

- 多画面预览
- 双码流切换
- 高分辨率回放
- 多路同步回放
- 抓图
- 导出
- 磁盘格式化
- AI 开关
- 音频对讲

### 8.2 需要把能力探测显式纳入动作链路

建议未来动作流都允许经过这样一段：

```text
UI Action
→ Capability Check
→ Controller Use Case
→ Service
→ HAL/SDK
→ State Update
→ UI Refresh
```

这条链路是你们项目和普通 UI 工程最大的差别之一。

### 8.3 需要给“降级执行”留结构位置

因为多 SoC 场景下，并不是所有动作都只有“成功/失败”两种结果。

很多情况下更现实的是：

- 能执行，但性能降级
- 能执行，但路数受限
- 能执行，但不能与其他动作并发
- 能执行，但仅某些通道支持

所以 004 要为后续架构设计明确提出：

**动作结果模型必须支持 degrade，不要只支持 pass/fail。**

## 9. 004 对 HAL 文档的接口分类输入

现在可以把 004 和 HAL 文档的关系直接固化下来。

### 9.1 Media HAL

承接：

- 预览
- 回放
- 抓图
- 录像
- 码流切换
- 显示绑定

### 9.2 Device HAL

承接：

- 通道信息
- 通道状态
- 设备参数
- 云台/音频等设备控制

### 9.3 Storage HAL

承接：

- 磁盘枚举
- 容量查询
- 格式化
- 录像文件管理
- 导出目标管理

### 9.4 System HAL

承接：

- 时间
- 网络
- 系统升级
- 重启
- 出厂恢复
- 系统状态获取

### 9.5 Capability HAL

承接：

- SoC/SDK 版本
- 预览路数能力
- 回放并发能力
- 编解码能力
- AI 能力
- 显示输出能力

这部分意味着：

004 现在已经可以直接作为 HAL 文档章节设计输入，而不是只做参考。

## 12. 最终建议

一句话概括：

**004 现在应该被定义为“交互结构化架构输入文档”，而不是“按钮关系整理说明”。**

我建议后续统一按这个认知推进：

- 页面不只是页面，要归入页面域
- 按钮不只是按钮，要归入动作域
- 动作必须关联状态域和服务域
- 多 SoC 能力差异必须显式进入动作链路
- HAL 分类应从这些动作自然反推

这样整理后的 004，才真正足够支撑 NVR 软件架构设计。
