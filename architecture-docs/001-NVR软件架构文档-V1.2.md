# 001《NVR 软件架构文档》V1.2

## 1. 文档目标

本文档用于把当前 NVR 项目的软件架构从“方向性说明”推进到“可落地设计骨架”，为后续详细设计、代码重构、HAL 接口抽取、页面交互实现、SoC 验证与下游交付提供统一依据。

V1.2 相比 V1.0 的重点提升有两点：

1. **补足落地粒度**
   - 不只说明分层方向，还补充核心模块职责边界、层间约束、关键流程、状态流转与重构切入点。
2. **补足架构逻辑清晰度**
   - 通过统一主链路、模块关系图和交互流说明，让架构不只“分了层”，而且“知道为什么这样流动、谁调用谁、状态在哪里收口”。

同时，本文档会吸收 MVP 架构中真正有价值的部分，但不会机械套用其教科书形式。

---

## 2. 本次迭代结论

当前 NVR 项目的正式架构，仍应采用：

**UI 展示层 + 页面控制/交互编排层 + 核心服务层 + 平台/HAL 适配层**

但在 V1.2 中，需要进一步明确：

1. **UI 不直接承接复杂逻辑**
2. **交互编排层负责把“用户动作”翻译成“系统用例”**
3. **核心服务层负责业务规则、状态一致性、资源生命周期与降级决策**
4. **HAL 层负责 SoC/SDK 差异收口，不向上暴露页面语义**
5. **系统以统一资源模型、统一状态模型、统一事件机制为主骨架**
6. **能力探测、功能裁剪、降级执行、诊断追踪必须进入主链路，而不是作为补丁机制存在**

## 3. MVP 可借鉴点与适用边界

MVP 对当前 NVR 架构最有价值的，不是命名，而是职责切分思想。

### 3.1 值得吸收的部分

#### 3.1.1 View 变薄

UI 页面、窗口、控件层应尽量只负责：

- 显示
- 输入收集
- 基础交互反馈
- 将用户动作上送给交互编排层

不应在页面里长期承载：

- 复杂业务判断
- 多步骤流程控制
- 媒体资源生命周期
- 能力差异判断
- 降级策略决策

#### 3.1.2 增加交互编排层

MVP 中 Presenter 的核心价值，在当前项目中更适合被扩展为：

**页面控制器 / 应用编排层（Controller / Presenter / UseCase Orchestrator）**

该层负责：

- 把按钮点击、页面进入、菜单选择翻译成系统动作
- 驱动跨服务流程
- 收敛异步回调
- 输出稳定 ViewState 给 UI

#### 3.1.3 页面状态显式化

不要让状态散落在：

- 页面成员变量
- 控件局部变量
- SDK 回调
- 定时器消息
- 临时全局变量

而要把页面需要消费的状态显式组织为：

- ViewState
- DomainState
- SessionState

#### 3.1.4 提高可测试性

交互编排层和服务层应能脱离具体 UI 运行，并可通过模拟依赖做行为验证。

### 3.2 不应照搬的部分

#### 3.2.1 不能把 Presenter 做成新上帝类

Presenter/Controller 只负责交互编排，不应吞掉：

- 业务规则
- 设备模型
- 底层媒体控制
- 存储管理
- 权限模型

#### 3.2.2 不能给每个页面都机械套三件套

复杂页面要补编排层，简单静态页面可以保持轻量。

#### 3.2.3 不能把领域逻辑抬到 UI 近侧

真正稳定的业务能力，仍然应沉淀到核心服务层。

## 4. 当前项目的正式架构主张

### 4.1 架构总原则

正式 NVR 系统不再以“页面”为主骨架，而以以下三类核心对象为骨架：

1. **资源对象**：设备、通道、会话、录像、存储卷、能力表
2. **状态对象**：预览状态、回放状态、配置状态、告警状态、能力状态
3. **事件对象**：设备事件、媒体事件、存储事件、配置事件、告警事件、诊断事件

页面只是这些对象的一个投影视图。

### 4.2 架构图（逻辑视图）

```text
┌──────────────────────────────────────────────────────────────┐
│                        UI 展示层                             │
│ Screen / Widget / Dialog / StatusBar / VideoViewport       │
│ 只负责显示、输入转发、局部交互反馈                           │
└──────────────────────────────────────────────────────────────┘
                            │ 用户动作 / ViewEvent
                            ▼
┌──────────────────────────────────────────────────────────────┐
│              页面控制 / 交互编排层（MVP 可借鉴核心）         │
│ PreviewController / PlaybackController / ConfigController   │
│ 职责：动作翻译、流程编排、异步收敛、ViewState 输出           │
└──────────────────────────────────────────────────────────────┘
            │ 调用 UseCase / Service        │ 订阅状态 / 事件
            ▼                               ▼
┌──────────────────────────────────────────────────────────────┐
│                        核心服务层                            │
│ Capability / Device / Preview / Playback / Record / Search │
│ Event / Storage / Config / Auth / Export / Diagnostic      │
│ 职责：业务规则、状态一致性、资源生命周期、降级决策           │
└──────────────────────────────────────────────────────────────┘
            │ 统一服务接口                 │ 发布 DomainEvent
            ▼                               ▼
┌──────────────────────────────────────────────────────────────┐
│                 领域模型 / 状态 / 事件骨架                   │
│ Model / State / Session / Event / Error / Result           │
│ 职责：统一对象语义，避免跨层各自拼字段                       │
└──────────────────────────────────────────────────────────────┘
            │ 依赖能力 / 资源访问
            ▼
┌──────────────────────────────────────────────────────────────┐
│                    平台 / HAL 适配层                         │
│ Media HAL / Display HAL / Storage HAL / System HAL         │

└──────────────────────────────────────────────────────────────┘
```

### 4.3 主链路解释

整个系统的主链路应固定为：

**用户动作 → Controller/Presenter 翻译 → Service 执行 → HAL 调用 → 状态/事件回收 → ViewState 输出 → UI 刷新**

这条链路必须稳定，不能在局部页面中随意打洞直连。

## 5. 分层职责与硬约束

## 5.1 UI 展示层

职责：

- 页面显示
- 控件交互
- 对话框/菜单
- 焦点、布局、主题、视觉反馈
- 消费 ViewState 并刷新 UI

禁止：

- 直接调用 HAL / SDK
- 自己维护复杂媒体生命周期
- 跨页面拼业务状态
- 自己做能力探测和降级判断

### 5.1.1 UI 输入输出规范

输入：
- ViewState
- ComponentState
- CapabilityHint
- NotificationModel

输出：
- ButtonClicked
- PageEntered
- MenuSelected
- SliderChanged
- DialogConfirmed

UI 只产生意图，不做复杂流程决策。

## 5.2 页面控制 / 交互编排层

职责：

- 接收 UI 动作
- 转换为系统用例
- 组织跨服务调用
- 收敛异步回调
- 把领域状态映射成 ViewState
- 决定 UI 级反馈策略（loading / disable / toast / dialog）

可借鉴 MVP 的点主要在这一层。

### 5.2.1 典型控制器划分

- PreviewController
- PlaybackController
- SearchController
- RecordController
- ConfigController
- EventController
- StorageController
- DiagnosticController

### 5.2.2 该层不应承担的职责

- 不直接持有底层句柄
- 不直接管理通道硬件资源
- 不定义底层错误码
- 不承载长期业务规则
- 不替代服务层做能力模型维护

## 5.3 核心服务层

职责：

- 提供稳定业务能力
- 维护正式状态机
- 管理资源生命周期
- 承接策略判断
- 对接事件中心与诊断中心

服务层是长期稳定骨架，不应被页面变化牵着走。

## 5.4 平台/HAL 适配层

职责：

- 封装私有 SDK
- 屏蔽 SoC 差异
- 输出统一能力查询接口
- 输出统一资源操作语义
- 输出统一错误码/状态码

限制：

- HAL 不感知页面
- HAL 不返回 UI 文案语义
- HAL 不参与页面级流程编排

## 6. 核心模块设计（V1.2 细化版）

以下模块开始从“说明职责”推进到“说明边界、输入输出、状态与依赖”。

### 6.1 CapabilityService

#### 职责

- SoC/SDK 能力探测
- 能力表归一化
- 功能开关决策支持
- 降级策略输入提供

#### 输入

- SoC 标识
- SDK 版本
- HAL 能力查询结果
- 编译期裁剪配置

#### 输出

- CapabilityProfile
- CapabilityState
- FeatureVisibilityDecision
- DegradeAdvice

#### 依赖

- Capability HAL
- Device HAL
- ConfigService（读取策略配置）

#### 关键状态

- unknown
- detecting
- detected
- partially_supported
- unsupported
- downgraded

#### 关键约束

- 所有 UI 和 Service 不得各自绕过它直接判断 SoC 能力
- 能力结果必须可缓存、可导出、可诊断

### 6.2 PreviewService

#### 职责

- 预览会话管理
- 画面与通道绑定
- 码流切换
- 多画面资源调度
- 预览降级控制

#### 输入

- channelId
- streamProfile
- layoutMode
- previewPolicy

#### 输出

- LiveSession
- PreviewState
- PreviewResult(success / fail / degraded)

#### 依赖

- CapabilityService
- DeviceService
- Media HAL
- Display HAL
- EventService
- DiagnosticService

#### 状态机

- idle
- starting
- active
- degraded
- failed
- stopping

#### 关键规则

- 单通道预览失败不应拖垮其他通道
- 多画面超能力上限时，优先执行降级，不直接整体失败
- 会话生命周期由服务层管理，不由页面对象管理

### 6.3 PlaybackService

#### 职责

- 回放会话管理
- 时间轴控制
- seek / pause / speed 控制
- 回放结果和状态一致性维护

#### 输入

- PlaybackSessionArgs
- timeRange
- speedMode
- channelSet

#### 输出

- PlaybackSession
- PlaybackState
- TimelineSnapshot

#### 依赖

- SearchService
- CapabilityService
- Media HAL
- Storage HAL
- DiagnosticService

#### 状态机

- idle
- loading
- ready
- playing
- paused
- seeking
- end_of_stream
- failed

### 6.4 ConfigService

#### 职责

- 配置加载、校验、保存
- 配置变更分发
- 配置生效协调
- 配置与运行态边界管理

#### 输入

- configSection
- configPayload
- validationPolicy

#### 输出

- ConfigSnapshot
- ConfigApplyResult
- ConfigState

#### 依赖

- Persistence
- EventService
- System HAL
- Device HAL

#### 关键规则

- 配置保存成功不等于配置已生效
- 必须区分：保存成功 / 生效成功 / 部分生效 / 生效失败

### 6.5 DiagnosticService

#### 职责

- 统一日志入口
- 操作链路追踪
- 版本/能力/性能/错误快照输出
- 为 SoC 验证提供诊断资料

#### 输入

- DomainEvent
- ErrorReport
- PerformanceMetric
- CapabilitySnapshot

#### 输出

- DiagnosticSnapshot
- TraceRecord
- ExportableLogBundle

#### 关键要求

- 所有关键链路必须带 traceId / actionId
- 必须可区分正常失败与降级成功

## 7. 层间接口约束

为避免后续实现走回页面驱动老路，V1.2 明确以下硬规则。

### 7.1 调用方向

允许：

- UI → Controller
- Controller → Service
- Service → HAL
- Service → Domain/Event/Infra

不允许：

- UI → HAL
- UI → SDK
- UI → 跨多个 Service 直接拼流程
- Controller → SDK
- HAL → UI

### 7.2 结果语义统一

关键服务接口统一返回以下结果语义之一：

- success
- fail
- degraded

不能只返回布尔值，否则无法表达多 SoC 降级运行。

### 7.3 错误语义统一

错误至少要区分：

- 用户输入错误
- 配置错误
- 资源不足
- 能力不足
- SDK/HAL 错误
- 系统异常
- 超时
- 状态冲突

### 7.4 状态所有权

- 页面显示状态归 Controller 输出
- 业务运行状态归 Service 持有
- 设备/底层资源状态归 HAL/Service 协同维护
- 能力状态归 CapabilityService 唯一维护

## 8. 关键流程设计

## 8.1 打开实时预览

### 主流程

1. UI 产生 `OpenPreview(channelId, targetViewport)`
2. PreviewController 检查当前页面上下文与 ViewState
3. PreviewController 调用 CapabilityService 查询当前能力
4. PreviewController 调用 PreviewService.startPreview(...)
5. PreviewService 完成资源申请、码流决策、显示绑定
6. Media HAL / Display HAL 执行底层启动
7. PreviewService 更新 PreviewState
8. Controller 将 DomainState 转换为 ViewState
9. UI 刷新画面、按钮状态、提示信息

### 失败与降级分支

- 若主码流不支持：切换到子码流，结果记为 degraded
- 若同屏路数超限：降低布局或只激活目标窗口，结果记为 degraded
- 若通道不可用：目标窗口进入 failed，不影响其他窗口

## 8.2 回放检索并开始播放

1. UI 发起 `SearchRecord(query)`
2. SearchController 调用 SearchService
3. SearchService 返回 SearchResult + SearchState
4. UI 选择目标结果
5. PlaybackController 组装 PlaybackSessionArgs
6. PlaybackService 结合 CapabilityService 与 Storage HAL 校验可播放性
7. Media HAL 建立回放会话
8. PlaybackState 进入 ready / playing
9. TimelineState 与 ViewState 同步更新

## 8.3 保存配置并生效

1. UI 发起 `SaveConfig(section, payload)`
2. ConfigController 进行基础交互校验
3. ConfigService 执行结构校验与持久化保存
4. ConfigService 决定是否立即生效、延迟生效或需重启生效
5. 相关 Service / HAL 接收配置变更事件
6. ConfigState 进入 applied / partial_applied / apply_failed
7. UI 展示结果与后续动作要求

## 8.4 存储异常处理

1. Storage HAL 上报异常事件
2. EventService / StorageService 接收并分类
3. DiagnosticService 记录 trace、错误码、上下文
4. 相关 Service 执行保护动作（暂停录像、限制导出、发出告警）
5. Controller 将状态映射到 ViewState
6. UI 展示异常提示与用户可执行动作

## 9. 统一状态模型（V1.2）

V1.2 不再只列状态枚举，而要求逐步建设状态机。

### 9.1 PreviewState

状态：
- idle
- starting
- active
- degraded
- failed
- stopping

触发：
- start request
- hal started
- degrade chosen
- start failed
- stop request
- resource released

关键迁移：
- idle → starting
- starting → active / degraded / failed
- active → stopping
- degraded → stopping
- stopping → idle

### 9.2 PlaybackState

状态：
- idle
- loading
- ready
- playing
- paused
- seeking
- end_of_stream
- failed

关键迁移：
- idle → loading
- loading → ready / failed
- ready → playing / paused
- playing → paused / seeking / end_of_stream / failed
- paused → playing / seeking / failed
- seeking → playing / paused / failed

### 9.3 ConfigState

状态：
- idle
- validating
- saving
- saved
- applying
- applied
- partial_applied
- apply_failed

这个状态对“保存成功但未生效”这种场景非常关键。

## 10. 统一事件机制

统一事件中心至少承载：

- DeviceEvent
- PreviewEvent
- PlaybackEvent
- RecordEvent
- SearchEvent
- AlarmEvent
- StorageEvent
- ConfigEvent
- CapabilityEvent
- DiagnosticEvent

### 10.1 事件使用原则

- 事件用于广播状态变化，不用于偷偷绕过服务边界
- 事件必须带来源模块、对象 ID、traceId、时间戳
- 关键故障与降级都要事件化，供 UI 与诊断消费

---

## 11. 能力探测、裁剪与降级闭环

V1.2 明确这不是附属机制，而是主设计的一部分。

### 11.1 启动阶段

系统启动后应按顺序完成：

1. SoC 识别
2. SDK 版本识别
3. HAL 能力查询
4. CapabilityProfile 归一化
5. 功能裁剪决策生成
6. 初始 UI 功能显示策略下发

### 11.2 运行阶段

每个关键动作执行前，应支持：

- 能力校验
- 资源校验
- 降级可行性判断
- 执行结果标记为 success / fail / degraded

### 11.3 UI 呈现阶段

UI 需要能区分：

- 不支持：隐藏或不可见
- 受限支持：可见但置灰/提示
- 降级成功：功能可用但能力受限
- 正常成功：完整能力可用

## 13. 重构落地路线图（V1.2）

## 13.1 第一批切口：先抽最容易绑死 UI 的链路

优先抽出：

- PreviewController + PreviewService
- PlaybackController + PlaybackService
- CapabilityService
- ConfigController + ConfigService

目标：

- 先把最复杂的交互流程从页面中抽出
- 建立第一批状态机和结果语义

## 13.2 第二批切口：统一状态和事件

优先统一：

- PreviewState
- PlaybackState
- ConfigState
- CapabilityState
- StorageState

并引入统一事件骨架。

## 13.3 第三批切口：SDK 调用下沉

目标：

- 把页面和零散业务代码中的 SDK 调用收口到 HAL
- 建立 SoC Adapter 结构

## 13.4 第四批切口：诊断链路成型

要求：

- 所有关键用例可追踪
- 所有失败和降级可区分
- 所有关键会话可导出上下文
